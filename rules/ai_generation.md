# AI 生成コードガイドライン

AI（CLINE 等）を使ったコード生成時のガイドラインです。

## 概要

AI によるコード生成を効果的に活用し、品質を保つためのガイドラインです。

---

## AI 活用の原則

1. **レビュー必須**: AI 生成コードは必ずレビュー
2. **理解してから使用**: 理解できないコードは使わない
3. **テスト作成**: 生成コードには必ずテストを追加
4. **段階的な生成**: 大きな機能は段階的に生成
5. **ドキュメント整合性**: 既存ドキュメントとの整合性確認

---

## CLINE 操作の効率化原則

AI エージェント（CLINE）を効率的に活用するための運用原則です。

### トークン節約

**目的**: API の利用制限に達しないように、トークン消費を最小限に抑える

- デバッグ対象のソースコードは一部のみを編集
- トークン制限を超えないよう、常に最適化を意識
- 「シンプル」という指示 = UI ではなく**コードの軽量化**を意味する

### ピンポイント編集

**目的**: 変更の影響範囲を最小化し、レビューを容易にする

- コード編集は**全ファイル再作成ではなく**、必要最小限の部分編集を行う
- 実績例: 全ファイル再作成（数万文字）→ ピンポイント編集（数百文字）
- `replace_in_file` ツールを積極的に活用

### 作業の細分化

**目的**: 文字数制限に達するリスクを回避

- 大きなタスクは小さなステップに分解
- 各ステップを完了してから次のステップへ進む
- 途中経過を確認しながら進行

### 進捗管理

**目的**: タスクの完遂を確実にする

- ステップバイステップで**25%ずつ**の進捗を報告
- 各マイルストーンのタスクを厳密に管理
- 完全かつ完璧な遂行を目指す

### 説明の明確性

**目的**: 変更の意図と影響を理解しやすくする

- **なぜそれをやるべきか**を常に説明
- 合理的かつわかりやすい説明を心がける
- 技術的な判断の根拠を明示

### 複数視点での検討

**目的**: より最適な解決策を導出する

- 5 人の CTO が議論・ディベートする想定で思考
- 複数の実装案を比較検討
- 最適解を選択する理由を明確にする

## CLINE の効果的な使い方

### 明確な指示

```
❌ 悪い例:
「ユーザー機能を作って」

✅ 良い例:
「ユーザー登録APIを作成してください。
- エンドポイント: POST /api/v1/users
- 入力: email, password, display_name
- バリデーション: メール形式チェック、パスワード8文字以上
- 出力: ユーザー情報とJWTトークン
- エラーハンドリング: メール重複時は409エラー」
```

### コンテキストの提供

- 関連ドキュメントを参照させる
- 既存のコードパターンを示す
- プロジェクトの規約を明示

## 生成コードのレビューポイント

### セキュリティ

- [ ] 入力バリデーション実装済み
- [ ] SQL インジェクション対策済み
- [ ] XSS 対策済み
- [ ] 認証・認可チェック実装済み

### パフォーマンス

- [ ] N+1 問題なし
- [ ] 適切なインデックス使用
- [ ] 不要なクエリなし

### コード品質

- [ ] 命名規則準拠
- [ ] コメント適切
- [ ] エラーハンドリング実装
- [ ] テストカバレッジ十分

---

## Checkpoint の活用

### タイミング

- 大きな機能実装前
- リファクタリング前
- 実験的な変更前

### 命名規則

```
checkpoint-名前
例: checkpoint-auth-implementation
```

---

## AI 生成の適用範囲

### 推奨

- ボイラープレートコード
- CRUD エンドポイント
- データモデル定義
- テストコード
- ドキュメント生成

### 慎重に

- セキュリティ関連
- 複雑なビジネスロジック
- パフォーマンスクリティカルな部分
